# Terminal + Copilot Dashboard Redesign — Implementation Plan

Produced 2026-07-03 by a 6-agent parallel research pass (each independently
reading the actual codebase, not the spec) + one synthesis pass, run via the
Workflow tool at the user's explicit request ("have a team of agents rigorously
plan"). Original task spec: rebuild `dashboard/src/app/page.tsx` (currently the
12-col "Command Deck" bento from PR #1085) into the "Home D · Terminal + Copilot"
mockup — a 7-pane mono-typeset terminal deck plus a feature-flagged copilot chat
drawer that proposes actions via cards a human must click ("chat proposes,
buttons dispose"). Full mockup: https://claude.ai/code/artifact/21d32382-dab6-4724-af35-da4d3270f19d
→ tab "Home / D · Terminal (dark)".

This document is a plan, not an implementation — nothing in it has been built
yet. See §6 for suggested PR sequencing and §7 for open questions that need a
human decision before work starts.

## 1. Executive summary

Phase 1 (the deck) is feasible largely as specced: 5 of 7 panes (attention, tail, fleet, next, vitals) map onto data plumbing and utilities that already exist in `page.tsx` and `dashboard/src/lib/`, and the cron-countdown pane's stated fallback ("minimal 5-field parser if nothing importable exists") turns out to be unnecessary — `dashboard/src/lib/humanSchedule.ts` already does this. Two panes (workers, inbox) are **not** reuse — `page.tsx` has zero existing fetch/state for either, and they require new proxy wiring plus (for workers) a status-word taxonomy that doesn't exist yet. Phase 2 (copilot) is feasible for three of four write paths, but the single biggest risk in the whole plan is that **the "chat proposes, buttons dispose" safety guarantee is not free** — it holds only if action cards call the existing gated handler functions (`handleToggleEnabled`, `handleRunClick`, `useRecipeInstall().handleInstall`) rather than the raw fetch/POST calls those handlers wrap, and for the **kill-switch specifically, no confirm-gate exists anywhere in the current UI to inherit** — that gap must be closed with a new shared confirm dialog before a copilot card is allowed to touch it. A second high-severity gap: the Decision Record (`ctxSaveTrace`/`DecisionTraceLog`) is MCP-only with no HTTP route and no `source` field, so "log every executed card action... attributable in /traces" as specced is not achievable without a schema change and a new bridge route — this needs its own small PR, not an incidental detail in the copilot PR.

### Pane 3 — next (cron countdown)
- **Spec assumption that doesn't hold, but favorably**: spec assumes "no cron-parse util importable, may need minimal 5-field parser." Research confirms: `node-cron` (root `package.json:139`) is Node-only server-side (`src/recipes/scheduler.ts:265,456-457`) and not usable from the Next.js client, but a **dependency-free, already-shipped, dashboard-local** parser exists: `dashboard/src/lib/humanSchedule.ts` — `humanizeSchedule()` (`:141`, returns `nextRunAt` epoch ms) and `describeNextRun()` (`:223`, "next in 14h" phrasing), covering `@every`/`@hourly`/`@daily`/`@weekly`/`@monthly`/`@yearly` macros + common 5-field cron shapes. It's intentionally total (never throws), falling back to raw string + `humanized:false` for exotic cron (steps/ranges/lists outside a leading `*/N` minute-step).
- **Reusable**: `/api/bridge/recipes` already returns `schedule?`, `trigger?`, `enabled?` per recipe (`recipes/page.tsx:158-163`, already read at `:1008`) — no new endpoint/field needed. `humanizeSchedule`/`describeNextRun` currently only consumed by the single-recipe detail page (`recipes/[...name]/page.tsx:52,700,746` via `dashboard/src/lib/recipeStatus.ts:26-28`).
- **Adjusted approach**: do **not** add a cron-parser dependency or hand-roll a new parser. Compose the existing utility: filter recipes to enabled + cron-triggered, map through `humanizeSchedule(r.schedule)`, sort by `nextRunAt` ascending, render via `describeNextRun()`. Recipes whose cron falls outside covered shapes get `nextRunAt: undefined` — pane must handle this gracefully (show raw schedule text, no countdown, or drop from the sorted list) rather than block on extending parser coverage. Only invest in extending `humanSchedule.ts`'s field coverage if real installed recipes are observed hitting the fallback path.
- **Tasks**: loop recipes list through `humanizeSchedule`/`describeNextRun` in the new pane; render mm:ss countdown (ticking 1s, client-only) below 1h else absolute time; include paused cron recipes greyed "(off)".

## 3. Phase 2 (copilot) — refined plan

### SAFETY FIRST — "chat proposes, buttons dispose" verdict

**Clear verdict, not hedged: three of the four write paths are safe to wire IF AND ONLY IF the action card calls the existing gated handler function, not the raw endpoint call it wraps. The fourth (kill-switch) is currently unsafe to wire at all — there is no confirm-gate anywhere in today's UI to inherit, so a copilot card would be exactly as reckless as a raw POST.**

- **Pause/enable recipe — safe via correct entry point.** `handleToggleEnabled` (`recipes/page.tsx:596-642`) PATCHes `/api/bridge/recipes/{name}` with optimistic flip + rollback, and **does fire `window.confirm` at `604-607` before disabling a non-manual-trigger recipe**. The copilot card must call `handleToggleEnabled(recipe)` itself (or an extracted equivalent) — not POST the endpoint directly — or the confirm dialog is silently skipped.
- **Run with vars — safe via correct entry point.** `handleRunClick` (`recipes/page.tsx:582-594`) owns the modal-vs-direct-run branch: any recipe with vars, or any recipe on the first run of a session (`hasConfirmedRunThisSession()`, sessionStorage `patchwork:run-confirmed`), always opens `RunModal` (`218-323`) before `executeRun` (`541-580`) fires. The card must call `handleRunClick`, not `executeRun` — calling `executeRun` directly would skip the vars form and the implicit session-confirm step.
- **Install from marketplace — safe via correct entry point, HIGH STAKES if wrong.** `useRecipeInstall` (`marketplace/page.tsx:251-305`) computes `elevated = requiresElevatedConfirm(recipe)`, which is **default-deny**: true unless `risk_level==='low' && network_access===false && file_access===false` (`registry.ts:49-55`) — and research confirms essentially every live-registry recipe today fails that bar, so almost all installs are elevated. `handleInstall()` (`287-293`) opens `InstallConfirmDialog` and returns *without* calling `runInstall()` when elevated; only non-elevated recipes skip straight to `runInstall`. **The copilot card must bind to the hook's returned `handleInstall`, never to `runInstall` or the page-level POST-triggering function directly** — a naive wiring that skips the hook's gate would silently bypass the confirm dialog for nearly every install. This is the single highest-stakes wiring mistake available in this feature.
- **Kill-switch engage/release — NOT safe today, gap must be closed first.** Both existing call sites (`KillSwitchBanner.tsx:21-45` release-only, and `settings/page.tsx:1122-1170` toggle for both engage/release) POST `/api/bridge/kill-switch` with **no client-side `window.confirm` at either site** — engage relies on nothing, release relies on nothing beyond a `locked` disabled-state. There is no existing gate for the copilot to inherit. **Do not ship a one-click kill-switch action card in the copilot drawer.** Either (a) add a shared confirm dialog to both `KillSwitchBanner` and the settings `ToggleRow` first (separate small PR, benefits the existing UI too), so the copilot inherits real friction, or (b) exclude kill-switch from Tier 1/2 lever matching in v1 and require the human to use the existing UI directly.
- **Action-card lifecycle pattern**: `ApprovalCard` (`approvals/page.tsx:187-540`) is a solid, directly reusable shape — local pending/executing state, `externalInFlight` merge for cross-surface triggers, tier-aware confirm *inside* the handler (not bypassed above it), toast on settle, expiry-timeout effect. Copilot cards should mirror this shape exactly.

### Tier 1 — levers (no LLM)
Intent-match against installed recipe/worker names for pause/enable/run/retry/kill-switch/open-page. Matched → emit action card directly, bound to the gated handlers above (`handleToggleEnabled`, `handleRunClick`, `useRecipeInstall().handleInstall`). Kill-switch excluded from Tier 1 v1 per the safety gap above unless the confirm-dialog PR lands first. Card shows action/target/trigger/reversibility, then result + trace ref post-execution (trace-ref availability blocked on the Decision Record HTTP gap — see below).

### Tier 2 — agent (LLM via existing orchestrator)
- **Confirmed real, cite the actual mechanism**: `runClaudeTask` (`src/tools/runClaudeTask.ts:14-95`) — `{prompt (≤32KB), contextFiles (≤20), timeoutMs, stream, model, effort, fallbackModel, maxBudgetUsd, ...}`; non-streaming returns `{taskId, status:'pending'}` immediately, streaming blocks with a progress callback. `getClaudeTaskStatus` (`src/tools/getClaudeTaskStatus.ts:5-115`) returns full status but **truncates inline output to 500 chars** in the MCP response — the fuller `task.output` (up to 50KB) is only exposed via the dashboard's own HTTP proxy (`/api/bridge/tasks`), which is what the existing `/tasks` page actually polls (`dashboard/src/app/tasks/page.tsx:362`), **not SSE** — adaptive 2s/10s interval poll, backing off once all tasks are terminal, paused on `document.hidden` (`354-401`), output shown as `task.output.slice(0, 8000)` (`:177`).
- **Decision**: the copilot drawer must decide which path it uses (MCP tool call vs the dashboard's HTTP `/api/bridge/tasks` proxy) since they return meaningfully different output sizes (500 chars vs up to 8000-char render). Recommend the HTTP proxy path, reusing the exact `/tasks` polling pattern — it's already adaptive and battle-tested; do not invent SSE streaming for this.
- **Driver detection, confirmed reusable**: `useBridgeStatus().patchwork?.driver` already carries the driver string (`useBridgeStatus.ts:26-33`, backed by `src/bridge.ts:1058,1741`). Gate Tier 2 UI on `driver` being a truthy non-`'none'` string; treat `undefined` (pre-first-poll or old bridge) the same as `'none'` — degrade silently to Tier-1-only, matching the existing `503 {unavailable:true}` pattern `/recipes/generate` already uses (`src/recipeRoutes.ts:1384-1395`).

### Recipe-creation flow
- `POST /recipes/generate` confirmed real and already dashboard-proxied (`src/recipeRoutes.ts:1329-1434`, proxy at `dashboard/src/app/api/bridge/recipes/generate`) — rate-limited, requires `--driver subprocess`, maps `errorKind` to real HTTP statuses.
- `POST /recipes/lint` confirmed real and dashboard-proxied (content-based, `src/recipeRoutes.ts:1719-1764`).
- **Gap**: `GET /recipes/doctor?recipe=<name>` is real and proxied but is **name-only** (already-installed recipes) — it is not a raw-YAML-content preflight check. The CLI's `recipe preflight <file.yaml>` verb has **no HTTP route at all** (zero grep hits for `/recipes/preflight`). So for a freshly-generated, not-yet-installed recipe, the only pre-save HTTP check available today is lint (syntax/best-practice), **not** the connector-authorization preflight the spec implies ("run lint + preflight on the result"). Flag explicitly rather than silently coding against an assumption: full preflight parity on a just-generated recipe requires a **new** HTTP route (e.g. `POST /recipes/preflight` accepting raw content) — scope as a small bridge-side PR, or descope Phase 2's recipe-creation card to lint-only + a note that full preflight happens post-install via the existing Doctor panel.
- **Tasks**: card renders YAML preview (existing `CodeBlock`) + lint badge (real) + preflight badge (either real, pending the new route, or omitted/marked "post-install" per the descope decision) + buttons `[Install (lands DISABLED)]` `[Edit YAML -> /recipes/new prefilled]` `[Discard]`. Install button must go through the recipe-install path with `enabled:false` forced.

### Worker-creation flow
Scaffold via the same `/recipes/generate` path with a worker template. Card must state proposed `autonomyCeiling` and never raise an existing one — this is a display/copy requirement on top of the generate response, no new plumbing beyond what recipe-creation already needs. Missing connector dependencies → "set up ->" link (reuse existing `/connections` deep link pattern).

### Chat-state and attribution
- Session-scoped chat state (memory + `sessionStorage`) — no bridge dependency, straightforward.
- **Missing — high risk**: "log every executed card action via the existing decision/trace write path... source: copilot tag if the shape allows" is **not currently possible without a schema change**. `ctxSaveTrace` (`src/tools/ctxSaveTrace.ts:20-73`) is MCP-tool-only; grepping `src/streamableHttp.ts` and `src/recipeRoutes.ts` for `ctxSaveTrace`/`/traces`/`recordDecision` returns zero route registrations. The backing `DecisionTrace`/`RecordDecisionInput` schema (`src/decisionTraceLog.ts:32-49,84-91`) has no `source`/`origin` field — the `origin: 'session'|'automation'` seen on Claude tasks is a *different* store (task-tracking, not the Decision Record) and doesn't help here.
- **Required before this requirement can ship**: (1) a new HTTP route exposing `ctxSaveTrace`-equivalent write access to the dashboard, (2) a schema extension adding a `source` field (or a synthetic sessionId sentinel like `dashboard-copilot`) threaded through `DecisionTraceLog.record()` and the JSONL persistence format. This touches a persisted, replay-relied-upon file (`decision_traces.jsonl`, consumed by `ctxQueryTraces` and the session-start digest) — treat as its own small PR, not a copilot-PR footnote.

## 4. Phase 3 (polish)

Marked optional in the spec; keep brief. Cmd+K binding, j/k tail scroll, and statusline segment links are all additive and low-risk once Phase 1/2 land — no new research risk surfaced here. The `useGlobalHotkey`/`usePaneShortcuts` extraction (see §5 sequencing) should happen *before* this phase adds a 7th copy-pasted keyboard `useEffect`, since the pattern is already duplicated 6x across the codebase (`recipes/page.tsx`, `tasks/page.tsx`, `runs/page.tsx`, `approvals/page.tsx`, `KeyboardShortcuts.tsx`, `useSearchHotkey.ts`) and is overdue for consolidation regardless of this feature.

## 6. Suggested sequencing

More granular than the spec's 3 phases, splitting out shared-logic extractions that multiple panes depend on so they land (and get tested) independently before the deck PR consumes them:

1. **PR 0 — docs/in-flight.md entry** (per repo convention), branch off main.
2. **PR 1 — extract `recipeRunHealth.ts`** (fleet pane data layer): pull `allRunsMap`/`successPct`/`avgDuration` out of `recipes/page.tsx` into `dashboard/src/lib/`, golden-master tests against current inline behavior first, reconcile with recipe-detail page if it duplicates the logic. No UI change.
3. **PR 2 — extract `workerTrust.ts`** (workers pane data layer): pull `primaryJob`/`isReversible`/`levelColor` + author new `deriveWorkerStatus()`, golden-master test against current `WorkerCard` output, get the "demoted" taxonomy decision resolved first (Open Questions). No UI change.
4. **PR 3 — shared `usePaneShortcuts`/`useGlobalHotkey` hook**: consolidate the 6x-duplicated tag/isContentEditable guard pattern into one hook, matching `useSearchHotkey.ts`'s shape. Small, independently useful.
5. **PR 4 — kill-switch confirm-dialog** (safety prerequisite for Phase 2): add shared confirm to `KillSwitchBanner` and settings `ToggleRow`. Ships value to existing UI regardless of copilot.
6. **PR 5 — Decision Record HTTP route + `source` field**: new route wrapping `ctxSaveTrace`-equivalent write, schema extension on `DecisionTrace`/`RecordDecisionInput`. Prerequisite for copilot action-attribution.
7. **PR 6 — Phase 1 deck** (the spec's Phase 1): 7-pane grid, statusline, consumes PR 1/2/3 outputs for panes 2/4, new proxy wiring for panes 4/6 inbox route, smoke test for `page.tsx` added here if not already present.
8. **PR 7 — Phase 2 copilot Tier 1 (levers only)**: feature-flagged, consumes PR 4/5, wired strictly through gated handlers per the safety section. Kill-switch lever only if PR 4 has landed.
9. **PR 8 — Phase 2 copilot Tier 2 (agent) + recipe/worker creation flows**: consumes existing `/recipes/generate`/`/recipes/lint`, descopes "preflight" to lint-only unless a `POST /recipes/preflight` route is built as a fold-in of this PR or a preceding one.
10. **PR 9 — Phase 3 polish** (optional): Cmd+K, j/k scroll, statusline links, `components/TerminalDeck/` extraction for `/deck` route.
