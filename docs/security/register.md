**Date:** 2026-07-06
**Type:** Consolidation + verification pass — **not a new audit.** This document merges every distinct finding raised across the five prior audits below into one register, then checks each one against the live code on `main` today (via direct file reads, `grep`, and `git log`) to determine whether it is actually still open, actually fixed, an explicit won't-fix, or could not be confirmed either way in the time available. Audit self-reported "fixed"/"open" labels were **not** trusted blindly in either direction — several findings marked "open" in 06-08/06-09 turned out to already be fixed on HEAD, and this pass surfaced a case (chained-runner negative retry) where a documented fix landed in one runner (flat/yamlRunner) but not its sibling (DAG/chainedRunner) — something a naive "check if the fix commit exists" pass would have missed.

## Summary

**134 distinct findings consolidated: 12 STILL-OPEN, 24 FIXED, 2 WONTFIX, 96 UNVERIFIABLE.**

Every HIGH-severity finding across all 5 docs, and the 5 explicitly-flagged candidate findings, were checked directly against current code in this pass (grep/Read, not trusting the source doc). Most MEDIUM findings and nearly all LOW findings are catalogued with title/severity/location/source-doc but marked UNVERIFIABLE — a fresh code read was not performed for each one given the volume (~250+ distinct items across all severities in the raw source docs before dedup). Duplicate findings across docs (e.g. the SSRF-DB-connector cluster, the postMessage wildcard-origin cluster, the token-exchange-body-leak cluster, the cron 5-vs-6-field cluster) are merged into one row with all source docs listed.

### The five flagged candidates

- **mcpClient init race** — **FIXED.** `src/connectors/mcpClient.ts:106-215` has `initInflight` promise-coalescing (`ensureInitialized()`), with an inline comment crediting "audit 2026-06-10 connectors-core-1" and citing `ChildBridgeClient.initInflight` as the mirrored pattern.
- **`parallel:{each}` no-op in chained recipes** — **FIXED (converted to a hard lint error, not a silent runtime no-op).** `src/recipes/validation.ts:219-246` now emits `chained-parallel-each-unsupported` when a chained recipe uses `parallel:{each}`, with a comment citing "M26" (the 06-19 finding ID). The failure mode moved from "silently throws on first run" to "caught at lint/preflight time" — a real fix, though technically it still *cannot run* rather than *running correctly*; flagging FIXED because the audit's stated harm (silent, undetected runtime failure) is eliminated.
- **Negative-retry handling** — **MIXED: FIXED in one runner, STILL-OPEN in the other.** `src/recipes/yamlRunner.ts:2131` clamps with `Math.max(0, step.retry ?? recipe.on_error?.retry ?? 0)` — fixed. `src/recipes/chainedRunner.ts:1470` still reads `const retryCount = step.retry ?? recipe.on_error?.retry ?? 0;` with **no clamp** — a negative `step.retry` on a chained/DAG recipe flows straight into `withRetry(..., retryCount, ...)` unguarded. The JSON schema (`schemas/recipe.v1.json:223`) does now declare `"minimum": 0` on `retry`, which blocks this at lint time for schema-validated recipes, but not for recipe objects constructed programmatically or replayed without going through the validator. Scored as **STILL-OPEN**, scoped specifically to the chained runner, in the table below.
- **Cron field validation** — **FIXED.** Both `src/recipes/scheduler.ts:449-458` (`/^\S+(?:\s+\S+){4,5}$/`) and `src/recipes/validation.ts:197-205` (`isCronExpr`, identical regex) now accept 5- or 6-field cron, with comments citing "audit 2026-06-09 sched-cron6-1" and "M24" respectively. Lint and runtime scheduler are consistent with each other now.
- **beta.12 npm tarball secret leak** — **STILL-OPEN / needs-user-action.** This item is not in any of the 5 audit docs read for this consolidation — it comes from project memory (`project_assessment_and_fixes_2026_06_24.md`) and was included here only because the task explicitly asked for it to be tracked. **No `npm` command of any kind (including read-only `npm view`) was run to check this** — per the task's explicit instruction not to attempt any npm command, this is reported as unverified-by-this-pass and flagged for direct user action: a human with npm publish credentials needs to confirm whether `npm deprecate patchwork-os@0.2.0-beta.12 "<reason>"` has been run, and run it if not.
