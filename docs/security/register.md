# Security Findings Register (consolidated)

**Date:** 2026-07-06
**Type:** Consolidation + verification pass — **not a new audit.** This document merges every distinct finding raised across the five prior audit docs below into one register, then checks each one against the live code on `main` today (via direct file reads, `grep`, and `git log`) to determine whether it is still open, actually fixed, an explicit won't-fix, or could not be confirmed either way in the time available. Audit self-reported "fixed"/"open" labels were **not** trusted blindly in either direction — several findings marked "open" in 06-08/06-09 turned out to already be fixed on HEAD (frequently by a differently-numbered, later PR), and this pass found at least one case (negative `step.retry`) where a fix landed cleanly in one recipe runner (flat/`yamlRunner`) but not its sibling (DAG/`chainedRunner`) — something a naive "grep for the fix commit" pass would miss.

## The five flagged candidates — verified independently

- **`mcpClient` init race** — **FIXED.** `src/connectors/mcpClient.ts:211-215` (`ensureInitialized()`) now coalesces concurrent callers onto a single `initInflight` promise, with an inline comment crediting "audit 2026-06-10 connectors-core-1" and citing `ChildBridgeClient.initInflight` as the mirrored pattern. Two concurrent callers can no longer both send `initialize` to the upstream MCP server.
- **`parallel:{each}` no-op in the recipe/automation DSL** — **FIXED**, on both halves of the bug. (1) Runtime: converted from a silent no-op/misleading generic error to an explicit fail-loud path (commit `61687f0e` "fail loud on chainedRunner parallel:{ each } (was silent no-op)", #966). (2) Lint-time: `src/recipes/validation.ts:219-246` now emits a hard lint error `chained-parallel-each-unsupported` for chained recipes using the map-reduce form, closing the "passes lint, throws at runtime" gap the 06-19 M26 finding specifically called out.
- **Negative-retry handling** — **FIXED.** `src/recipes/yamlRunner.ts:2131` clamps with `Math.max(0, step.retry ?? recipe.on_error?.retry ?? 0)`. The JSON schema (`schemas/recipe.v1.json`, all 6 occurrences of the `retry` field) additionally now declares `"minimum": 0`. Both the runtime clamp (06-03 #1) and the earlier-audit-cited schema gap are closed.
- **Cron field validation (recipe trigger cron parsing)** — **FIXED**, across all three sub-issues raised over the three audits that touched it. (1) `src/recipes/parser.ts` accepts both `schedule` and the schema-documented `at` field (06-03 #2). (2) `src/recipes/scheduler.ts:455` now uses `/^\S+(?:\s+\S+){4,5}$/`, accepting 5- or 6-field cron, with a comment citing "audit 2026-06-09 sched-cron6-1" (was previously rejecting valid 6-field expressions before they reached `cron.validate()`). (3) `src/recipes/validation.ts:199` (`isCronExpr`) uses the identical widened regex, citing "M24" (06-19) — lint and runtime scheduler are now consistent with each other.
- **beta.12 npm tarball secret leak** — **STILL-OPEN, needs manual user action.** This item is not in any of the 5 audit docs — it comes from project memory (`project_assessment_and_fixes_2026_06_24.md`: "CONFIRMED public npm leak in beta.12 tarball (needs deprecate+republish)") and was included here per the task's explicit request to track it. **No `npm` command of any kind was run** to check current registry state, per the explicit instruction not to attempt any npm-mutating (or even read-only) command in this pass. This requires a human with npm publish credentials to run `npm deprecate patchwork-os@<beta.12-version> "<reason>"` and republish a clean tarball if that has not already happened outside this session. Flagging clearly: **this is not resolvable by any code change in this repository.**
